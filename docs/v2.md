# Upgrading to StreamSockets 2.0.0

This guide is for **operators** running StreamSockets 1.7.0 (or earlier) who want to upgrade to **2.0.0**. If you're reading the source and want the architecture of the rewrite itself, see [`/MIGRATION.md`](../MIGRATION.md) instead.

---

## TL;DR

- **2.0.0 is a Rust rewrite.** Same product (UDP-over-WebSocket tunnel), faster runtime, fixed reconnect bugs, native `wss://`, `io_uring` on Linux.
- **It is a breaking wire-protocol change.** v1.7.0 clients **cannot** talk to v2.0.0 servers, and vice versa. You must upgrade clients and servers together — or run them on separate endpoints during the transition.
- **Your `accounts.yml` is unchanged.** Same file, same schema, no edits needed.
- **Your Prometheus dashboards are unchanged.** All v1 metric names + labels are preserved; new ones are additive.
- **One env var is removed**: `USE_OLD_PROTOCOL` (no longer meaningful).
- **Three response status codes changed**: `401` for IP-denied / route-denied / reuse-conflict has split into `401` (token unknown) / `403` (IP or route denied) / `409` (account in use).

If you are deploying with **Docker Compose**, jump to [§4](#4-docker-compose-upgrade). Running on **systemd**, jump to [§5](#5-systemd-upgrade). Running on **Kubernetes**, jump to [§6](#6-kubernetes-upgrade).

---

> **⚠️ Security note for operators using `CLIENT_IP_HEADER`.**
> If your server runs behind an L4/L7 proxy that injects `X-Forwarded-For` (or
> a similar header named via `CLIENT_IP_HEADER`), set `CLIENT_IP_HEADER_TRUSTED_CIDRS`
> to the comma-separated CIDRs of those proxies. Without it, any client that
> can reach the server directly (e.g., misconfigured firewall, sidecar misroute)
> can inject the header and impersonate any IP in `allowedIps`. The server emits
> a startup warning when `CLIENT_IP_HEADER` is set without an allowlist; treat
> it as a hard misconfiguration in production.
>
> Example: `CLIENT_IP_HEADER=X-Forwarded-For` and `CLIENT_IP_HEADER_TRUSTED_CIDRS=10.0.0.0/8,fd00::/8`.

## 1. What changed

### 1.1 Breaking changes

| Area | v1.7.0 behavior | v2.0.0 behavior |
|---|---|---|
| Wire protocol | Two paths: header-based (`X-Route-Address`+`X-Route-Port`) **and** legacy `X-Auth-Route` + JSON `TextWebSocketFrame` route negotiation. | Header-only. JSON path is gone. v1 clients get HTTP `400` from a v2 server. |
| Client env var | `USE_OLD_PROTOCOL=true` opted into the legacy JSON path. | **`USE_OLD_PROTOCOL` is removed.** Setting it is ignored with a warn log; the v2 client only speaks v2. |
| Server response codes | `401` covered: token unknown, IP not allowed, route not allowed. | `401` = token unknown. `403` = IP not allowed *or* route not allowed. `409` = account in use (`reuse: false`). `400` = malformed (missing required header). |
| Server-side TLS | Relied on a terminating proxy (HAProxy/Nginx) for `wss://`. | Native `rustls` termination — set `TLS_CERT_FILE` + `TLS_KEY_FILE` on the server. |
| Text frames after handshake | Logged and ignored (latent bug). | Connection closed with WebSocket close code `1003 Unsupported Data` and metric `streamsockets_protocol_violations_total{reason="text_frame"}`. |
| Multi-source UDP on the client | Each new UDP source spawned a fresh WebSocket (race-prone, source of several bugs). | Single-source lock-on: the first UDP source is locked for the lifetime of the client process; subsequent foreign sources are dropped + counted (`streamsockets_client_foreign_sources_total`). Run multiple client processes/containers if you need multi-source. |
| Default backoff initial delay | 1 s | 100 ms (honors `RETRY_INITIAL_DELAY_SECONDS` when ≥ 1 s, else clamps at 100 ms floor). |
| Backoff jitter | None (deterministic 1, 2, 4, 8, 16, 30 …). | Decorrelated jitter — eliminates thundering-herd reconnect storms. |
| Ping-timeout detection latency | ~15 s (1 s checker × 5-miss rule). | ~25 s with defaults; tunable down to ~6 s with `PING_INTERVAL_MILLIS=2000 PING_TIMEOUT_MILLIS=4000 PING_PERMITTED_MISSES=0`. |

### 1.2 What did **not** change

- `accounts.yml` schema — same file works on v1 and v2.
- Default ports — server 8080/9090, client UDP 9000.
- Header names: `X-Auth-Type`, `X-Auth-Token`, `X-Route-Address`, `X-Route-Port`, `CLIENT_IP_HEADER` semantics.
- Prometheus metric names + labels (preserved 1:1; new metrics additive).
- Docker image entrypoint, working directory, healthcheck endpoints.
- All env vars except `USE_OLD_PROTOCOL` (see [§7](#7-env-var-changes-quick-reference)).

### 1.3 New features worth knowing about

- **Native `wss://`** without a terminating proxy.
- **`/healthz`, `/livez`, `/readyz`** on the metrics port (`9090`) — k8s-friendly.
- **PROXY-protocol-v2** support (optional) for L4 load balancers that need to preserve the real client IP without an `X-Forwarded-For` header.
- **Graceful drain on `SIGTERM`**: `/healthz` flips to 503, in-flight tunnels finish, hard-close after `SHUTDOWN_GRACE_SECONDS=30`.
- **systemd `Type=notify` + watchdog** support out of the box.
- **A bunch of new metrics** for reconnect / queue / handshake observability — see [§8](#8-new-metrics).

## 2. Pre-flight checklist

Run through this **before** scheduling the upgrade window:

- [ ] You are on v1.7.0. (Earlier versions: upgrade to v1.7.0 first, then to v2.0.0. v2 only validates against v1.7.0 wire compat.)
- [ ] You have an inventory of every place `WEBSOCKET_URI` is set on clients (Docker Compose, systemd unit files, Helm values, Ansible inventory, …). Each one must be flipped to the v2 server endpoint when its host upgrades.
- [ ] You have shell or kubectl access to all your StreamSockets clients. Cutover requires a binary swap **and** a config change in the same deploy.
- [ ] Your Prometheus is scraping `streamsockets_*` metrics today. Grafana dashboards should keep working unmodified — verify post-upgrade.
- [ ] You've reviewed the response-code change. If you have any custom monitoring or scripts that match specifically on HTTP `401` from the StreamSockets server, update them to also match on `403` (IP/route denied) and `409` (reuse conflict).
- [ ] You have a rollback plan. Keep the v1.7.0 image tag (`hyperxpro/streamsockets:server-1.7.0`, `hyperxpro/streamsockets:client-1.7.0`) pulled and ready on every host.
- [ ] (If running Linux): you've reviewed [§9 sysctl recommendations](#9-recommended-sysctls). Apply them ahead of the upgrade so the new server has them on first boot.

## 3. Recommended upgrade pattern

> **Why two endpoints and not in-place?** v1 and v2 cannot interoperate on the same TCP connection. If you flip a single endpoint from v1 to v2 without coordinating clients, **every v1 client connecting at the moment of cutover gets HTTP `400` and goes into retry-loop hell**. Run two endpoints during the migration window so traffic shifts cleanly.

The recommended path:

1. Stand up v2.0.0 server on a **second endpoint** (different port, different hostname, or both) next to your existing v1.7.0 server.
   - Both servers mount the same `accounts.yml`.
   - Both target the same upstream(s).
   - Both scrape into the same Prometheus.
2. Migrate clients in batches by changing each client's `WEBSOCKET_URI` to the v2 endpoint **and** swapping its image to `client-2.0.0` in the same deploy. Watch the new metrics ([§8](#8-new-metrics)) for the migrated batch.
3. Once 100% of clients are on v2 and stable for ≥ 7 days, decommission the v1 server.

A typical timeline:

| Day | Action |
|---|---|
| T-7 | v2 server stood up on `:8081` (or `tunnel-v2.example.com`); 0 % traffic; smoke tests pass. |
| T 0 | Migrate canary 5 % of clients (one container or one systemd unit at a time). Monitor for 24h. |
| T+1 | Ramp to 25 %. |
| T+3 | 50 %. |
| T+7 | 100 % of clients on v2. v1 server idle. |
| T+14 | Decommission v1 server. |

## 4-pre. Plain Docker upgrade (without Compose)

If you run the containers via raw `docker run` rather than `docker compose`,
the equivalent procedure is:

```bash
# 1. Pull v2.0.0 images on every host that runs server or client
docker pull hyperxpro/streamsockets:server-2.0.0
docker pull hyperxpro/streamsockets:client-2.0.0

# 2. Start the v2 server alongside the v1 server on a different port
docker run -d --name streamsockets-server-v2 \
  --restart unless-stopped \
  -p 8081:8081/tcp -p 9091:9091/tcp \
  -v "$PWD/accounts.yml:/etc/streamsockets/accounts.yml:ro" \
  -e ACCOUNTS_CONFIG_FILE=/etc/streamsockets/accounts.yml \
  -e BIND_PORT=8081 \
  -e METRICS_PORT=9091 \
  -e LOG_FORMAT=json \
  -e DISABLE_IOURING=true \
  hyperxpro/streamsockets:server-2.0.0

# 3. Per client host: stop the v1 client, start the v2 client pointed at v2 endpoint
docker stop streamsockets-client && docker rm streamsockets-client
docker run -d --name streamsockets-client \
  --restart unless-stopped \
  -p 9000:9000/udp \
  -e WEBSOCKET_URI=ws://your-server:8081/tunnel \
  -e AUTH_TOKEN=secret \
  -e ROUTE=example.com:8888 \
  -e EXIT_ON_FAILURE=true \
  hyperxpro/streamsockets:client-2.0.0
```

Confirm:

```bash
docker logs -f streamsockets-server-v2     # expect "all tunnel listeners pre-bound"
curl -s http://localhost:9091/readyz        # expect 200 OK once the workers spawn
docker logs -f streamsockets-client         # expect "websocket live" within ~1s
```

Rollback per host: stop the v2 client, re-run the v1 client with the v1
`WEBSOCKET_URI`. **Do not flip only the URI** — a v2 client cannot speak
to a v1 server.

## 4. Docker Compose upgrade

### 4.1 Server side

Add a v2 service alongside the existing v1 service. **Do not edit the v1 service** during the migration window.

```yaml
services:
  streamsockets-server:                    # v1.7.0 — keep running unchanged
    image: hyperxpro/streamsockets:server-1.7.0
    container_name: streamsockets-server
    restart: unless-stopped
    environment:
      - ACCOUNTS_CONFIG_FILE=/app/accounts.yml
      - BIND_ADDRESS=0.0.0.0
      - BIND_PORT=8080
      - WS_PATH=/tunnel
      - METRICS_PORT=9090
    volumes:
      - ./accounts.yml:/app/accounts.yml
    ports:
      - "8080:8080/tcp"
      - "9090:9090/tcp"

  streamsockets-server-v2:                 # v2.0.0 — new
    image: hyperxpro/streamsockets:server-2.0.0
    container_name: streamsockets-server-v2
    restart: unless-stopped
    environment:
      - ACCOUNTS_CONFIG_FILE=/app/accounts.yml
      - BIND_ADDRESS=0.0.0.0
      - BIND_PORT=8081                     # different port from v1
      - WS_PATH=/tunnel
      - METRICS_PORT=9091                  # different metrics port too
      - LOG_FORMAT=json
      - DISABLE_IOURING=true               # default in image; flip if your seccomp allows io_uring
    volumes:
      - ./accounts.yml:/app/accounts.yml
    ports:
      - "8081:8081/tcp"
      - "9091:9091/tcp"
```

Bring up the new service:

```bash
docker compose up -d streamsockets-server-v2
docker compose logs -f streamsockets-server-v2          # confirm "WebSocketServer started on 0.0.0.0:8081"
curl -s http://localhost:9091/healthz                   # expect "OK"
curl -s http://localhost:9091/metrics | head            # expect streamsockets_runtime_kind set
```

### 4.2 Client side (per host)

For each client, in its `compose.yml`, change **two** things in the same edit:

```diff
 services:
   streamsockets-client:
-    image: hyperxpro/streamsockets:client-1.7.0
+    image: hyperxpro/streamsockets:client-2.0.0
     environment:
-      - WEBSOCKET_URI=ws://your-server:8080/tunnel
+      - WEBSOCKET_URI=ws://your-server:8081/tunnel
       - AUTH_TOKEN=secret
       - ROUTE=example.com:8888
-      - USE_OLD_PROTOCOL=false        # remove — env var no longer exists
```

Apply:

```bash
docker compose up -d streamsockets-client
docker compose logs -f streamsockets-client          # confirm "Live" state in the FSM logs
```

If you keep `USE_OLD_PROTOCOL` set on a v2 client, it's ignored with a warn log on startup. Delete it for cleanliness.

### 4.3 Decommissioning v1 (after 100% client migration)

```bash
docker compose stop streamsockets-server
docker compose rm -f streamsockets-server
# update compose.yml: delete the v1 service block, rename v2 → drop the "-v2" suffix if you want
```

## 5. systemd upgrade

### 5.1 Server side

Install the v2 binary alongside v1 — different unit name, different port:

```bash
# Build the v2 binary (or download from releases)
sudo install -o root -g root -m 0755 streamsockets-server-2.0.0 /usr/local/bin/streamsockets-server-v2
```

Create `/etc/systemd/system/streamsockets-server-v2.service`:

```ini
[Unit]
Description=StreamSockets Server v2.0.0
After=network-online.target
Wants=network-online.target

[Service]
Type=notify
User=streamsockets
Group=streamsockets
ExecStart=/usr/local/bin/streamsockets-server-v2
Restart=always
RestartSec=2
WatchdogSec=30
LimitNOFILE=1048576
LimitNPROC=65536

# Hardening
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ProtectHome=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectControlGroups=true
RestrictNamespaces=true
RestrictRealtime=true
LockPersonality=true
MemoryDenyWriteExecute=true
SystemCallArchitectures=native
# If you want io_uring (and your kernel + seccomp allow it), include the next line.
# Otherwise leave it commented out and set DISABLE_IOURING=true below.
SystemCallFilter=@system-service @io-event io_uring_setup io_uring_enter io_uring_register

Environment="ACCOUNTS_CONFIG_FILE=/etc/streamsockets/accounts.yml"
Environment="BIND_ADDRESS=0.0.0.0"
Environment="BIND_PORT=8081"
Environment="METRICS_PORT=9091"
Environment="LOG_FORMAT=json"
Environment="DISABLE_IOURING=false"

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now streamsockets-server-v2
sudo systemctl status streamsockets-server-v2
sudo journalctl -u streamsockets-server-v2 -f
```

The v1 server unit (`streamsockets-server.service`) keeps running on port 8080 untouched.

### 5.2 Client side (per host)

If you have a single client unit on each host:

```bash
sudo install -o root -g root -m 0755 streamsockets-client-2.0.0 /usr/local/bin/streamsockets-client
sudo systemctl edit streamsockets-client                # add or override:
#   [Service]
#   Environment="WEBSOCKET_URI=ws://your-server:8081/tunnel"
#   # remove any USE_OLD_PROTOCOL line
sudo systemctl restart streamsockets-client
sudo journalctl -u streamsockets-client -f
```

If you want a `Type=notify` upgrade for the client unit too (recommended), update the `[Service]` block to `Type=notify` and add `WatchdogSec=30`. The v2 binary calls `sd_notify(READY=1)` on first `Live` transition.

### 5.3 Decommissioning v1 server

```bash
sudo systemctl stop streamsockets-server
sudo systemctl disable streamsockets-server
sudo rm /etc/systemd/system/streamsockets-server.service
sudo systemctl daemon-reload
```

## 6. Kubernetes upgrade

### 6.1 Two Deployments + two Services

`server-v2-deployment.yaml`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: streamsockets-server-v2
spec:
  replicas: 3
  selector:
    matchLabels: { app: streamsockets-server, version: v2 }
  template:
    metadata:
      labels: { app: streamsockets-server, version: v2 }
    spec:
      terminationGracePeriodSeconds: 45     # SHUTDOWN_GRACE_SECONDS + buffer
      containers:
      - name: server
        image: hyperxpro/streamsockets:server-2.0.0
        ports:
        - { containerPort: 8080, name: tunnel }
        - { containerPort: 9090, name: metrics }
        env:
        - { name: BIND_ADDRESS, value: "0.0.0.0" }
        - { name: BIND_PORT, value: "8080" }
        - { name: METRICS_PORT, value: "9090" }
        - { name: LOG_FORMAT, value: "json" }
        - { name: DISABLE_IOURING, value: "true" }       # cluster default seccomp blocks io_uring
        - { name: SHUTDOWN_GRACE_SECONDS, value: "30" }
        readinessProbe:
          httpGet: { path: /readyz, port: metrics }
          periodSeconds: 5
        livenessProbe:
          httpGet: { path: /livez, port: metrics }
          periodSeconds: 10
        startupProbe:
          httpGet: { path: /healthz, port: metrics }
          failureThreshold: 30
          periodSeconds: 2
        resources:
          requests: { cpu: "1", memory: "256Mi" }
          limits:   { cpu: "4", memory: "1Gi" }
        volumeMounts:
        - { name: accounts, mountPath: /etc/streamsockets, readOnly: true }
      volumes:
      - name: accounts
        configMap: { name: streamsockets-accounts }
---
apiVersion: v1
kind: Service
metadata:
  name: streamsockets-server-v2
spec:
  type: ClusterIP
  selector: { app: streamsockets-server, version: v2 }
  ports:
  - { name: tunnel, port: 8080, targetPort: tunnel }
  - { name: metrics, port: 9090, targetPort: metrics }
```

The v1 Deployment + Service stay in place under different labels (`version: v1`).

### 6.2 Client cutover

For each Deployment / DaemonSet / StatefulSet that runs the client, update the image **and** the `WEBSOCKET_URI` env in a single rollout:

```bash
kubectl set image deploy/your-game-server streamsockets-client=hyperxpro/streamsockets:client-2.0.0 \
  --record
kubectl set env deploy/your-game-server -c streamsockets-client \
  WEBSOCKET_URI=ws://streamsockets-server-v2:8080/tunnel \
  USE_OLD_PROTOCOL-                       # delete the env var
kubectl rollout status deploy/your-game-server
```

The trailing `-` after `USE_OLD_PROTOCOL` is `kubectl set env` syntax for "unset this env var."

### 6.3 Decommissioning v1

```bash
kubectl delete deployment streamsockets-server     # the v1 one
kubectl delete service streamsockets-server
```

## 7. Env var changes (quick reference)

### 7.1 Removed

| Env var | Where | What to do |
|---|---|---|
| `USE_OLD_PROTOCOL` | client | Delete it. Setting it on a v2 client logs a warn on startup; the value is ignored. |

### 7.2 New (server)

| Env var | Default | Purpose |
|---|---|---|
| `PROXY_PROTOCOL` | `disabled` | `disabled` \| `optional` \| `v1` \| `v2`. Enables PROXY-protocol parsing on accept (preserves real client IP under L4 LB). |
| `MAX_CONCURRENT_CONNECTIONS` | `100000` | Cap concurrent tunnels. `0` = unlimited. Returns `503` + `Retry-After: 5` once reached. |
| `SHUTDOWN_GRACE_SECONDS` | `30` | Time to drain in-flight tunnels on `SIGTERM` before hard-close. |
| `LOG_FORMAT` | `json` (Docker) / `pretty` (TTY) | `json` \| `pretty` \| `compact` |
| `LOG_LEVEL` | `info` | `tracing` env-filter syntax (`info,streamsockets=debug` works). |
| `TLS_CERT_FILE`, `TLS_KEY_FILE` | unset | Enables native `wss://`. Both must be set together. |
| `TLS_REQUIRE_CLIENT_CERT` | `false` | Require client certificate (mTLS). When `true`, set `TLS_CLIENT_CA_FILE` to a PEM bundle of trusted issuers. |
| `TLS_CLIENT_CA_FILE` | unset | PEM bundle of trusted client-cert issuers. Required when `TLS_REQUIRE_CLIENT_CERT=true`. |
| `CLIENT_IP_HEADER_TRUSTED_CIDRS` | unset | **Security-critical.** Comma-separated CIDR allowlist of trusted L4/L7 proxies. When set, `CLIENT_IP_HEADER` is honored ONLY for direct connections from these CIDRs. **If unset and `CLIENT_IP_HEADER` is set, any direct attacker can spoof the header to bypass `allowedIps`.** A startup warning fires in that configuration; set this var (e.g. `10.0.0.0/8,192.168.0.0/16`) to silence it. |
| `TOKIO_CONSOLE_BIND` | unset | Binds `tokio-console` for prod debugging. Localhost-only by default. |

### 7.3 New (client)

| Env var | Default | Purpose |
|---|---|---|
| `PING_PERMITTED_MISSES` | `1` | Extra missed pongs tolerated before tear-down. Combined with `PING_INTERVAL_MILLIS` and `PING_TIMEOUT_MILLIS`, controls dead-peer detection latency. |
| `RETRY_BUDGET` | `100` | Consecutive failed connects before terminal exit. |
| `QUEUE_MAX_BYTES` | `1048576` | Reconnect-queue cap (drop-oldest when full). |
| `QUEUE_DRAIN_TIMEOUT_MS` | `30000` | Purge the reconnect queue after this much sustained `Reconnecting` time. |
| `UDP_CHANNEL_CAPACITY` | `1024` | Bounded MPSC capacity between the UDP listener and the FSM live loop. Frames over this cap are dropped + counted in `streamsockets_queue_dropped_total{reason="channel_full"}` rather than allowed to grow unbounded. |
| `LOG_FORMAT`, `LOG_LEVEL` | as server | |

### 7.4 Unchanged (preserved 1:1)

Server: `ACCOUNTS_CONFIG_FILE`, `CLIENT_IP_HEADER`, `PARENT_THREADS`, `CHILD_THREADS`, `BIND_ADDRESS`, `BIND_PORT`, `HTTP_MAX_CONTENT_LENGTH`, `WS_PATH`, `MAX_FRAME_SIZE`, `METRICS_ENABLED`, `METRICS_BIND_ADDRESS`, `METRICS_PORT`, `METRICS_PATH`, `ACCOUNTS_RELOAD_INTERVAL_SECONDS`, `DISABLE_IOURING`.

Client: `THREADS`, `BIND_ADDRESS`, `BIND_PORT`, `WEBSOCKET_URI`, `AUTH_TOKEN`, `ROUTE`, `PING_INTERVAL_MILLIS`, `PING_TIMEOUT_MILLIS`, `RETRY_INITIAL_DELAY_SECONDS`, `RETRY_MAX_DELAY_SECONDS`, `UDP_TIMEOUT`, `EXIT_ON_FAILURE`, `DISABLE_IOURING`.

## 8. New metrics

All preserved v1 metric names + labels keep working. Existing dashboards do not need edits. The additions worth wiring into Grafana are below; all stay in the `streamsockets_*` namespace.

| Metric | Type | Labels | Useful alert |
|---|---|---|---|
| `streamsockets_handshake_failures_total` | counter | `reason` | rate > 0.1/s sustained ⇒ misconfigured client or attack |
| `streamsockets_reconnect_attempts_total` | counter | `account_name`, `result` | sudden spike ⇒ network event or server flap |
| `streamsockets_reconnect_state` | gauge | `account_name` | `2` (terminal) on any account ⇒ page |
| `streamsockets_ping_timeouts_total` | counter | `account_name` | rising rate per account ⇒ tunnel health degrading |
| `streamsockets_ping_rtt_seconds` | histogram | `account_name` | p99 jump ⇒ network change |
| `streamsockets_queue_dropped_total` | counter | `account_name`, `reason` | rate > 0 ⇒ tunnel can't keep up; check upstream |
| `streamsockets_queue_purged_total` | counter | `account_name` | rate > 0 ⇒ sustained reconnect failure |
| `streamsockets_protocol_violations_total` | counter | `reason` | non-zero ⇒ misbehaving client (or stale v1 client trying to talk to v2 server) |
| `streamsockets_upstream_dns_failures_total` | counter | — | rate > 0 sustained ⇒ DNS issue |
| `streamsockets_upstream_connect_failures_total` | counter | — | rate > 0 ⇒ upstream down |
| `streamsockets_ws_close_total` | counter | `side`, `code` | `code=1011` rate spike ⇒ server-side errors |
| `streamsockets_handshake_version` | counter | `version` | exposes whether your fleet is fully migrated |
| `streamsockets_runtime_kind` | gauge | `kind` | set once at startup; `io_uring` vs `epoll` vs `tokio` |
| `streamsockets_active_tunnels` | gauge | — | server-wide capacity gauge |

A starter Prometheus alert rule set (Alertmanager-friendly):

```yaml
groups:
- name: streamsockets
  rules:
  - alert: StreamSocketsHandshakeFailureRate
    expr: rate(streamsockets_handshake_failures_total[5m]) > 0.1
    for: 10m
    labels: { severity: warning }
  - alert: StreamSocketsClientTerminal
    expr: streamsockets_reconnect_state == 2
    for: 1m
    labels: { severity: critical }
  - alert: StreamSocketsQueueDropping
    expr: rate(streamsockets_queue_dropped_total[5m]) > 0
    for: 5m
    labels: { severity: warning }
  - alert: StreamSocketsProtocolViolation
    expr: rate(streamsockets_protocol_violations_total[5m]) > 0
    for: 1m
    labels: { severity: warning }
    annotations:
      description: "Likely a v1 client connecting to a v2 server. Check {{ $labels.reason }}."
```

## 9. Recommended sysctls

Apply these on every host running a v2 server. They are not strictly required to start, but throughput and tail latency at scale depend on them.

```bash
# /etc/sysctl.d/99-streamsockets.conf
net.core.rmem_max            = 33554432
net.core.wmem_max            = 33554432
net.core.rmem_default        = 4194304
net.core.wmem_default        = 4194304
net.core.netdev_max_backlog  = 10000
net.core.somaxconn           = 8192

# UDP global memory (low / pressure / max in 4-KiB pages)
net.ipv4.udp_mem             = 12148128 16197504 24296256

# Critical for our UDP-tunnel workload
net.netfilter.nf_conntrack_udp_timeout         = 60
net.netfilter.nf_conntrack_udp_timeout_stream  = 600
net.netfilter.nf_conntrack_max                 = 1048576

net.ipv4.ip_local_port_range = 1024 65535
net.ipv4.tcp_tw_reuse        = 1
fs.nr_open                   = 1048576
```

```bash
sudo sysctl --system
```

The most important one is `nf_conntrack_udp_timeout_stream=600` (default is 180 s, which silently reaps idle tunnels behind NAT after 3 min). If you only change one thing, change this.

## 10. Load balancer settings

If you run an LB in front of the server, raise WS-friendly timeouts. The PING interval defaults to 5 s; the LB idle timeout must be at least 2× that, ideally 10×.

| LB | Setting | Recommended value |
|---|---|---|
| AWS NLB | idle timeout | `350s` (max) |
| AWS ALB | idle timeout | `4000s` (target group must be HTTP/1.1) |
| GCP TCP LB | timeout | `86400s` |
| HAProxy | `timeout tunnel` | `1h` minimum |
| HAProxy | `timeout client` / `timeout server` | `30s` (handshake-only — `tunnel` takes over after upgrade) |
| Nginx | `proxy_read_timeout` | `3600s` |
| Nginx | `proxy_buffering` (for `/tunnel`) | `off` |
| Envoy | cluster `idle_timeout` | `3600s` |
| Envoy | `upgrade_configs` | `websocket` |
| Envoy | route `timeout` | `0s` (disable per-request timeout for WS routes) |

If your LB sits between clients and the v2 server and you want the server to see real client IPs, set `PROXY_PROTOCOL=v2` on the server **and** enable PROXY v2 emission on the LB (HAProxy `send-proxy-v2` on the backend, AWS NLB target-group `proxy_protocol_v2.enabled=true`, Envoy `transport_socket: envoy.transport_sockets.upstream_proxy_protocol`). Without PROXY, the server will use the TCP peer address (the LB's address) as the client IP, which breaks `allowedIps` CIDR enforcement.

## 11. Rollback

If the v2 deployment misbehaves:

### 11.1 Server-side rollback

Because v2 runs on a separate endpoint, **rolling back is just stopping the v2 server** and pointing migrated clients back at the v1 endpoint.

```bash
# Docker Compose
docker compose stop streamsockets-server-v2

# systemd
sudo systemctl stop streamsockets-server-v2
sudo systemctl disable streamsockets-server-v2

# Kubernetes
kubectl scale deploy/streamsockets-server-v2 --replicas=0
```

### 11.2 Client-side rollback (per host)

Each migrated client must be reverted to v1 in two steps in the same change: image-tag back to v1, `WEBSOCKET_URI` back to the v1 endpoint. **Do not just flip the URI** — a v2 client cannot speak to a v1 server.

```bash
# Docker Compose
sed -i 's/client-2.0.0/client-1.7.0/' compose.yml
sed -i 's/:8081/:8080/'                compose.yml
docker compose up -d streamsockets-client

# systemd
sudo systemctl edit streamsockets-client       # revert WEBSOCKET_URI
sudo install -o root -g root -m 0755 streamsockets-client-1.7.0 /usr/local/bin/streamsockets-client
sudo systemctl restart streamsockets-client

# Kubernetes
kubectl rollout undo deploy/your-game-server   # if the previous revision had v1 image + URI
```

### 11.3 No data migration ever needed

`accounts.yml` is unchanged. Prometheus metrics are unchanged in name/label. There is nothing to "migrate back."

## 11.5 Release validation (run before any production cutover)

Before you flip a single production client to v2, the release must pass:

| Gate | Where it runs | Pass criteria |
|---|---|---|
| `cargo test --workspace --release --locked` | CI (`rust.yml`) | All tests pass on `1.86` and `stable` toolchains. |
| `cargo clippy --all-targets --all-features -- -D warnings -W clippy::pedantic` | CI | Zero warnings. |
| `cargo deny check` | CI | No advisories, no banned crates, no license violations. |
| `cargo audit` (weekly cron) | CI | No known vulnerabilities in `Cargo.lock`. |
| 5-minute FD-leak smoke (`SOAK_DURATION_HOURS=0.083`) | CI on every release tag | No FD growth over 5 min × 1000 tunnels. |
| **24-hour FD-leak soak** (`SOAK_DURATION_HOURS=24`) | **operator-run on representative hardware** | No FD growth over baseline; no memory growth; no metric drift. Spec §11.3. |

The 24-hour soak runs from `streamsockets-testsuite/tests/soak_fd_leak.rs`,
marked `#[ignore]` so it never runs in regular CI. Operator command:

```bash
SOAK_DURATION_HOURS=24 \
cargo test --release --test soak_fd_leak \
  -- --ignored --nocapture
```

The test reports `/proc/self/fd` count every 60 s and asserts no growth
beyond the baseline window. If FD count rises monotonically by more than
2 × N (where N = 1000 tunnels), **do not ship**: investigate the leak in
the per-tunnel cleanup path before any client cutover. The most common
cause is a tokio task missing a `select!` arm on `shutdown.cancelled()`,
leaving the task alive past its tunnel.

## 12. Troubleshooting / FAQ

### My v1.7.0 client keeps logging "Failed to parse JSON response" / "received 400 Bad Request" against the new server.

This is expected. v2 servers don't speak v1's JSON route negotiation. Either repoint the client at the v1 server (during migration) or upgrade the client to `client-2.0.0`.

### I upgraded a client to v2 and it can't connect — the v1 server is still up.

Yes — v2 client cannot speak to a v1 server (it sends v2-only headers, and v1 server expects either v2 headers *or* `X-Auth-Route` JSON; the v2 client only sends the former, and on a v1 server with mismatched route shape the handshake will fail). Make sure `WEBSOCKET_URI` points at the **v2** server endpoint.

### I'm seeing `streamsockets_handshake_failures_total{reason="bad_request"}` increment.

Most likely cause: a v1 client is hitting a v2 server. Find the client (the failure log includes peer IP) and either upgrade it or repoint it.

### My alert "401 from StreamSockets" stopped firing — is auth still working?

`401` only fires now when the **token** is unknown. The previous "IP not allowed" and "route not allowed" cases now return `403`. Update your alerts to match `401|403|409`.

### I'm running behind cgroup-v1 / older kernel — can I still use io_uring?

If your kernel is < 6.0 or your seccomp profile blocks `io_uring_setup`, the v2 binary detects this and falls back to epoll automatically. Look for `runtime=epoll reason=...` in the startup log. You can force this with `DISABLE_IOURING=true`. Performance is still excellent on epoll.

### The `streamsockets_runtime_kind` metric shows `tokio` — is that bad?

Not necessarily, but on a supported deployment it's a misconfiguration signal. Production targets are Linux x86_64 / aarch64 only, where you'd see `io_uring` (kernel ≥ 6.0, no seccomp block) or `epoll` (anything else). The `tokio` label only appears on unsupported non-Linux dev builds.

### Multi-source UDP on the client doesn't work like it used to.

Correct — v2 client locks onto the **first** UDP source for the lifetime of the process. Subsequent foreign sources are dropped + counted (`streamsockets_client_foreign_sources_total`). If you need multi-source, run multiple client containers/units, each binding a different UDP listener. The v1 multi-source path was the source of three of the seven reconnect bugs we're fixing.

### My ping-timeout detection feels different.

Yes. v1 took ~15 s to detect a dead peer (1 s checker × 5-miss rule). v2 with defaults takes ~25 s but is single-deadline (no polling). Tune down with `PING_INTERVAL_MILLIS=2000 PING_TIMEOUT_MILLIS=4000 PING_PERMITTED_MISSES=0` for ~6 s detection. Don't go below `PING_INTERVAL_MILLIS=500` — you'll start getting false positives on transient packet loss.

### My reconnect rate metric shows higher numbers immediately after upgrade.

v2 emits `streamsockets_reconnect_attempts_total` per attempt — v1 didn't have this metric, so the rate was invisible. This is observability working, not a regression. Compare the rate against your baseline of "tunnel uptime" instead.

### Do I need to change my `accounts.yml`?

No. It's byte-compatible. Same fields, same shape, same hot-reload behavior.

### How long will the v1.7.0 image / Java source stay around?

The v1.7.0 Docker images (`hyperxpro/streamsockets:server-1.7.0` / `:client-1.7.0`) stay published on Docker Hub indefinitely as a rollback target. The Java source has been removed from the repo as of the post-cutover restructure; if you need to rebuild from source, check out the `Release v1.7.0` git tag.
